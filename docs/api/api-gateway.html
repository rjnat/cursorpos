The API Gateway serves as the single entry point for all client requests to the CursorPOS microservices ecosystem. It provides:

Key Features

  • Authentication & Authorization - JWT token validation with defense-in-depth approach
  • Request Routing - Intelligent routing to appropriate microservices

üîê Authentication

Defense-in-Depth Security

The API Gateway implements a defense-in-depth approach:

  1. Gateway Validation - First layer validates JWT token and extracts claims
  2. Header Propagation - Tenant context forwarded via headers (X-Tenant-Id, X-User-Id, X-User-Role)
  3. Service Re-validation - Each microservice independently re-validates the JWT for autonomy

JWT Token Structure

JWT tokens must include the following claims:

{
  "sub": "user-uuid",           // User ID
  "tenant_id": "tenant-xxx",    // Tenant ID (required)
  "role": "ADMIN",              // User role
  "store_id": "store-yyy",      // Store ID (optional)
  "branch_id": "branch-zzz",    // Branch ID (optional)
  "permissions": ["..."],       // User permissions
  "exp": 1700000000,            // Expiration timestamp
  "iat": 1699999000             // Issued at timestamp
}

Public Endpoints (No Authentication Required)

  • POST /api/v1/auth/login - User login
  • POST /api/v1/auth/register - User registration
  • POST /api/v1/auth/refresh - Token refresh
  • POST /api/v1/auth/forgot-password - Password reset request
  • POST /api/v1/tenants/signup - Tenant self-service signup
  • GET /actuator/health - Health check
  • GET /actuator/info - Service information

Protected Endpoints

All other endpoints require a valid JWT token in the Authorization header:

Authorization: Bearer <jwt-token>

Propagated Headers (Gateway ‚Üí Services)

The gateway automatically extracts JWT claims and adds these headers for downstream services:

Header Source Description
X-Tenant-Id JWT tenant_id claim Current tenant context (required)
X-User-Id JWT sub claim Authenticated user ID
X-User-Role JWT role claim User's primary role
X-Store-Id JWT store_id claim Current store context (if applicable)
X-Branch-Id JWT branch_id claim Current branch context (if applicable)
Note: Services can trust these headers as they are set by the gateway after successful JWT validation. However, services also re-validate the JWT token independently for defense-in-depth security.

‚ö†Ô∏è Error Responses

Authentication Errors

401 Unauthorized

Returned when authentication fails or token is invalid/expired.

{
  "success": false,
  "message": "Invalid or expired token",
  "errorCode": "UNAUTHORIZED"
}

Common Causes:

  • Missing Authorization header
  • Malformed Authorization header (not starting with "Bearer ")
  • Expired JWT token
  • Invalid JWT signature
  • Token missing required claims (especially tenant_id)
404 Not Found

Route not found or service unavailable.

{
  "timestamp": "2025-11-22T10:30:00.000+00:00",
  "path": "/api/v1/unknown",
  "status": 404,
  "error": "Not Found",
  "message": "No matching route found"
}
503 Service Unavailable

Downstream service is unavailable (circuit breaker open).

{
  "timestamp": "2025-11-22T10:30:00.000+00:00",
  "status": 503,
  "error": "Service Unavailable",
  "message": "Service temporarily unavailable"
}
429 Too Many Requests

Rate limit exceeded.

{
  "timestamp": "2025-11-22T10:30:00.000+00:00",
  "status": 429,
  "error": "Too Many Requests",
  "message": "Rate limit exceeded. Please try again later."
}

‚öôÔ∏è Gateway Configuration

Rate Limiting

Redis-based rate limiting is configured per service route:

Service Replenish Rate Burst Capacity Description
Identity Service 10 req/sec 20 requests Standard rate for auth operations
Admin Service 10 req/sec 20 requests Standard rate for admin operations
Product Service 20 req/sec 40 requests Higher rate for frequent catalog queries
Transaction Service 20 req/sec 40 requests Higher rate for POS operations

Circuit Breaker Configuration

Resilience4j circuit breakers protect against cascading failures:

  • Sliding Window Size: 10 requests
  • Minimum Calls: 5 (before calculating failure rate)
  • Failure Rate Threshold: 50%
  • Wait Duration in Open State: 5 seconds
  • Half-Open Permitted Calls: 3

Retry Configuration

Automatic retries for transient failures:

  • Max Retries: 3
  • Retry Statuses: BAD_GATEWAY, GATEWAY_TIMEOUT
  • Retry Methods: GET, POST
  • Backoff Strategy: Exponential (10ms ‚Üí 50ms)

CORS Configuration

Cross-Origin Resource Sharing settings:

  • Allowed Origins: http://localhost:5173, http://localhost:5174, http://localhost:5175
  • Allowed Methods: GET, POST, PUT, DELETE, PATCH, OPTIONS
  • Allowed Headers: All (*)
  • Allow Credentials: true
  • Max Age: 3600 seconds

üí° Best Practices

Security

  • Always include the JWT token in the Authorization header for protected endpoints
  • Tokens should be securely stored (e.g., httpOnly cookies, secure storage)
  • Implement token refresh mechanism before expiration
  • Never log or expose JWT tokens in error messages
  • Validate tenant_id claim is present in all business operations

Performance

  • Implement client-side caching for frequently accessed data
  • Use appropriate HTTP methods (GET for read, POST for create, etc.)
  • Handle rate limiting gracefully with exponential backoff
  • Monitor circuit breaker status for service health
  • Keep connections persistent when making multiple requests

Error Handling

  • Always check response status codes before processing
  • Implement proper error handling for 401 (refresh token) and 403 (forbidden)
  • Handle 503 errors as temporary service unavailability
  • Log error responses for debugging
  • Provide user-friendly error messages in UI

üß™ Testing the Gateway

Using cURL

# Health Check
curl http://localhost:8080/actuator/health

# Login (Public Endpoint)
curl -X POST http://localhost:8080/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"user@example.com","password":"password123"}'

# Access Protected Endpoint
curl http://localhost:8080/api/v1/products \
  -H "Authorization: Bearer YOUR_JWT_TOKEN_HERE"

# Access with Full Headers
curl http://localhost:8080/api/v1/transactions \
  -H "Authorization: Bearer YOUR_JWT_TOKEN" \
  -H "Content-Type: application/json"

Using Postman

  1. Set request method and URL (e.g., GET http://localhost:8080/api/v1/products)
  2. Go to "Authorization" tab, select "Bearer Token"
  3. Paste your JWT token
  4. Add Content-Type: application/json to Headers if sending body
  5. Send request
